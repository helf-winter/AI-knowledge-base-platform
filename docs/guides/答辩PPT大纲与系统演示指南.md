# AI 企业知识库管理平台答辩 PPT 大纲与系统演示指南

本文档用于课程答辩、PPT 制作和现场系统演示。内容按照答辩时的讲述顺序组织，覆盖系统功能完成情况、待完善内容、PPT 大纲、演示路线和具体验证方法。

## 1. 项目定位

本项目是一个面向企业内部知识流转场景的 AI 知识库管理平台。

它不是单纯的“文档上传工具”，也不是单纯的“AI 聊天机器人”，而是围绕课题要求中的三条主线展开：

1. 知识生产：用户、文档和 AI 都可以产生知识。
2. 知识管理：知识需要被分类、检索、审核、发布、隔离和治理。
3. 知识消费：用户可以通过 AI 问答、专家 Agent 和 Skill 能力使用知识。

系统的核心亮点是把“上传文档 -> 解析知识 -> 权限隔离 -> AI 问答 -> 反馈评价 -> 自动学习 -> 审核发布 -> 公有知识复用”串成闭环。

## 3. 与课题要求的对应关系

### 3.1 知识的生产

课题要求：支持多种方式产生知识，例如用户手动录入、文档导入、AI 自动提炼。

本系统已实现：

- 文档上传导入。
- 手动创建个人知识。
- AI 问答补充知识。
- 自动学习发现知识缺口。
- AI 生成待审核知识草稿。
- 知识整合 Agent 生成相似知识合并草稿。

答辩表述建议：

> 我们把知识生产分成三类：用户主动生产、文档导入生产、AI 使用过程中被动发现和提炼。AI 生成的内容不会直接进入公有知识库，而是先作为个人知识或审核草稿，保证企业知识的准确性和可治理性。

### 3.2 知识的管理

课题要求：支持知识组织、分类、检索、基础管理和状态区分。

本系统已实现：

- personal / public / department 知识空间。
- owner_id 个人归属。
- 知识类别、标签、工作类别、文件类型等筛选。
- 知识条目的 reviewing / available / archived 等状态。
- 发布审核、访问审核、公有知识建议审核、知识整合审核。
- 操作审计和 Trace ID。

答辩表述建议：

> 企业知识不能只解决“能不能搜到”，还要解决“谁能看、谁能改、谁能发布、发布后如何追踪”。所以系统引入了知识空间、审核流、权限过滤和审计日志。

### 3.3 知识的消费

课题要求：基于知识库提供智能服务能力，可以是专家 Agent，也可以是 Skill。

本系统已实现：

- AI 问答助手。
- 专家 Agent 创建。
- 专家配置：名称、职责、知识范围、技能组合、回答边界。
- Skill 能力目录：知识检索、文档总结、流程提取、知识对比、知识缺口发现、发布草稿生成。
- 回答来源引用和文档跳转。

答辩表述建议：

> 知识消费不是让用户自己打开文档慢慢找，而是把知识封装为智能问答、专家 Agent 和可复用 Skill。这样知识从静态文档变成了可对话、可追踪、可治理的智能服务。

## 4. PPT 制作大纲

建议控制在 12 到 15 页。

### 第 1 页：标题页

标题：AI 企业知识库管理平台

副标题：面向企业知识生产、管理、消费与自动学习闭环

可放内容：

- 项目名称
- 团队/姓名
- 技术栈简写：Next.js + FastAPI + PostgreSQL + pgvector + DeepSeek + BGE-m3

### 第 2 页：业务背景与痛点

讲述重点：

- 企业知识分散在文档、对话、经验和制度中。
- 普通知识库只能存文档，不能持续学习。
- AI 问答如果缺少知识治理，容易胡编或泄露权限。

建议图示：

```text
分散文档 / 员工经验 / 问答记录 / 制度流程
        ↓
难检索、难复用、难更新、难审核
        ↓
需要 AI 知识管理平台
```

### 第 3 页：课题要求拆解

按三条主线讲：

- 知识生产：文档导入、手动录入、AI 提炼。
- 知识管理：分类、检索、权限、审核、状态。
- 知识消费：AI 问答、专家 Agent、Skill。

再补充加分项：

- 自动学习闭环。
- 多模态录入。
- 热度统计。
- 一键专家 Agent。

### 第 4 页：系统总体架构

建议图示：

```text
用户浏览器
  ↓
Next.js 前端
  ↓
FastAPI 后端
  ├─ Auth / 权限控制
  ├─ Knowledge Service
  ├─ Search / RAG
  ├─ Agent / Skill
  ├─ Review / Audit
  └─ Task / Auto Learning
  ↓
PostgreSQL + pgvector
  ├─ documents
  ├─ chunks
  ├─ embeddings
  ├─ access_requests
  ├─ publish_requests
  ├─ conversations
  └─ audit_logs
```

讲述重点：

- Next.js 负责页面和交互。
- FastAPI 是后端业务服务。
- PostgreSQL 存业务数据。
- pgvector 存语义向量。
- DeepSeek 负责生成式回答和审核建议。
- BGE-m3 负责本地语义向量生成。

### 第 5 页：知识生产流程

建议图示：

```text
文档上传
  ↓
后台解析
  ↓
Chunk 切分
  ↓
BGE-m3 向量化
  ↓
写入 PostgreSQL / pgvector
  ↓
进入个人知识空间
```

同时说明：

- 用户也可以手动创建知识。
- AI 问答补充也会形成个人知识。
- 大文件进入后台任务，避免阻塞页面。

### 第 6 页：知识空间与权限隔离

讲述重点：

- personal：个人知识，只能本人访问。
- public：公有知识，经过审核后可被符合条件的用户检索。
- department：部门知识，按部门或工作类别访问。
- 管理员不能随意查看用户个人知识，只能在用户提交审核后查看必要内容。

建议图示：

```text
用户 A 个人知识  ── 仅用户 A
用户 B 个人知识  ── 仅用户 B
提交发布审核
  ↓ 管理员审核
公有知识引用 ── 符合权限的用户可检索
```

### 第 7 页：知识检索能力

讲述重点：

- 关键词检索。
- BGE-m3 语义检索。
- pgvector 向量相似度。
- 标签筛选。
- 权限过滤。
- 轻量 rerank。

建议图示：

```text
用户问题
  ↓
语义召回 + 关键词召回
  ↓
权限过滤
  ↓
标签/类别/空间筛选
  ↓
重排序
  ↓
返回可引用知识片段
```

### 第 8 页：AI 问答助手

讲述重点：

- 用户不用选择“通用模式/知识库模式”。
- 系统优先知识库回答。
- 知识不足时自动通用兜底。
- 回答结束后可确认扩充知识。
- 支持多轮上下文和用户隔离缓存。

### 第 9 页：专家 Agent 与 Skill

讲述重点：

- 选择知识范围。
- 自动生成专家名称、职责、适用问题、禁止回答范围。
- 绑定 Skill 能力。
- 专家回答时优先使用绑定知识范围。

Skill 示例：

- 查找制度依据。
- 总结文档。
- 提取流程步骤。
- 对比知识差异。
- 发现知识缺口。
- 生成发布草稿。

### 第 10 页：审核与知识治理

讲述重点：

- 文档访问申请审核。
- 个人知识发布审核。
- 公有知识修改建议处理。
- AI 辅助审核只给建议，不自动审批。
- DeepSeek 不可用时有规则 fallback。

建议图示：

```text
用户提交申请
  ↓
AI 给出建议、风险等级、理由
  ↓
管理员人工通过/拒绝
  ↓
系统更新授权或发布状态
```

### 第 11 页：自动学习闭环

讲述重点：

- 低置信度回答。
- 用户负反馈。
- 高频问题。
- 进入知识缺口池。
- AI 生成草稿。
- 管理员审核发布。
- 后续问答命中更准。

建议图示：

```text
问答失败/低置信度/用户反馈
  ↓
知识缺口
  ↓
AI 草稿
  ↓
管理员审核
  ↓
公有知识
  ↓
更好的问答
```

### 第 12 页：知识整合 Agent

讲述重点：

- 企业知识会重复，例如“VPN 使用说明”“AI 补充 VPN”“VPN 申请经验”。
- 知识整合 Agent 扫描相似文档。
- 生成合并建议。
- 审核通过后创建整合知识文档。
- 原始文档保留，不直接删除。

### 第 13 页：工程实现与部署

讲述重点：

- 前后端分离。
- Alembic 数据库迁移。
- Docker Compose 部署。
- 一键启动脚本。
- 测试验证：后端编译、单测、前端构建。

操作：

讲述：

操作：

3. 在回答末尾选择“扩充”。
4. 回到知识库查看新增或追加的个人知识。

验证点：

- 未审核前其他用户搜不到。
- 审核通过后符合权限的用户能搜到。
- 审核拒绝后仍然保持个人知识。

5. 管理员手动通过或拒绝。

讲述：

> AI 在这里不是自动审批者，而是审核助手。最终决策由管理员完成。

验证点：

验证点：

## 8. 答辩常见问题与回答

### Q1：这个系统和普通 RAG 问答有什么区别？

答：

普通 RAG 主要解决“问问题、找文档、生成答案”。本系统在此基础上增加了企业知识治理：个人/公有/部门知识隔离、访问申请、发布审核、AI 辅助审核、问答反馈、自动学习、知识整合和审计追踪。也就是说，它不仅能回答问题，还能持续沉淀和治理知识。

### Q2：AI 生成的知识谁来保证准确？

答：

系统设计上不允许 AI 自动发布公有知识。AI 只负责发现缺口和生成草稿，最终必须由管理员审核。个人问答补充默认进入个人知识空间，不直接影响全公司。

### Q3：管理员是否能看到所有个人知识？

答：

不能。管理员可以管理公有知识和审核申请，但不能随意查看用户未提交审核的个人知识。只有用户主动提交发布审核时，管理员才可以看到必要的审核内容。

### Q4：是否支持真实语义检索？

答：

支持。当前已接入本地 BGE-m3 embedding，使用 pgvector 存储向量并进行相似度检索。同时保留关键词召回作为补充。

### Q5：为什么还需要标签筛选？

答：

语义检索解决“意思相近”，标签筛选解决“范围约束”。企业场景中用户经常需要限定知识类别、知识空间、文件类型或适用人员，所以两者需要结合。

### Q6：大文件怎么处理？

答：

大文件进入后台解析任务，不阻塞上传请求。页面展示排队、解析、失败等状态，失败后可以手动重试或继续解析。已解析的 chunk 和 embedding 会尽量复用。

### Q7：DeepSeek 不可用怎么办？

答：

核心流程不会中断。问答、审核、自动学习等模块都有规则 fallback，DeepSeek 不可用时仍能给出基础结果或草稿。

## 9. 推荐演示话术

开场：

> 我们做的是一个企业内部 AI 知识库管理平台。它围绕知识生产、知识管理和知识消费三个方向展开，并额外实现了自动学习闭环和专家 Agent 能力。

讲系统亮点：

> 这个系统的重点不是简单上传文档后问答，而是解决企业知识能不能被安全复用、能不能持续更新、AI 生成内容能不能被审核治理的问题。

讲权限：

> 用户上传的内容默认是个人知识，不会直接进入公有知识库。要变成全公司可用知识，必须提交发布审核。管理员也不能随意查看用户个人知识。

讲 AI：

> AI 在系统里不是唯一决策者，而是辅助者。它可以回答问题、发现缺口、生成草稿、给出审核建议，但最终发布和审批都由人来完成。

总结：

> 最终形成的是一个知识飞轮：用户使用知识，系统发现缺口，AI 生成草稿，管理员审核发布，知识库变得更完整，后续问答效果继续提升。
